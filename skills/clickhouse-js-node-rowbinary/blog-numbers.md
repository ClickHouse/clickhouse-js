**Benchmark environment**
Apple M4 Max · Node v24.6.0 · macOS 26.5.1 · `@clickhouse/rowbinary` @ `199f994` (main HEAD, v0.2.0 in-repo source) · ClickHouse server 26.1.1.200 · 50k rows/schema via `npm run bench` (vitest/tinybench).

**1. RowBinary vs JSON on numeric-heavy schemas.** Comparing `JSONEachRow` (V8 `JSON.parse`) against the generic library-API RowBinary reader on identical result sets, RowBinary wins by **~2.5–3.3x** on a wide-integer financial ledger (`UInt128`/`Int64`/`Decimal128`/`UInt256`) and a stable **~2.1x** on a dense Float/Int IoT schema. The ledger figure is hardware-sensitive — **3.3x** on an Apple M4 Max, **2.5x** on a 4-core AMD EPYC cloud runner ([public CI run](https://github.com/ClickHouse/clickhouse-js/actions/runs/28439460847), with a downloadable JSON artifact) — so cite the range, not a single number. The ledger number is also measured against the _correct_ JSON path: bare `JSONEachRow` is only ~1.8x slower but **silently rounds every wide value to float64** — you have to quote server-side and re-parse to bigint by hand to get a correct decode. _Flag, as expected:_ on a string-heavy log schema JSON **wins** (JSONCompactEachRow beats even the optimized RowBinary reader), which is exactly the skill's "don't use RowBinary here" guidance.

**2. Monomorphic vs naive parser.** Against a naive parser built by composing the library's public per-type read functions in a loop, the agent-inlined monomorphic parser is **~1.5–2.5x faster** on numeric-heavy schemas (ledger 1.55x, IoT 2.46x), holding ≥1.5x across three schemas (ledger, IoT, orders 3.4x). _Flag:_ the ~1.5x does **not** reproduce on a flat all-scalar row (`UInt64, String, DateTime`) — it's ~1.05x, because with no hot per-row formatter and no composite structure, monomorphization has nothing to remove. There the simpler API reader is the right call.

**3. Per-parser generation cost** (Claude Sonnet 4.6, skill loaded, averaged over 4 numeric schemas): **~230k tokens in** (mostly prompt-cache re-reads across the agent loop; the unique skill footprint read is ~28k), **~2.1k tokens out**, **~$0.20 per parser** with prompt caching (~$0.72 as an uncached upper bound).

**The UUID trap — a closed-book coin flip.** RowBinary stores a UUID as two little-endian `UInt64` halves, each byte-reversed relative to the text form. Asked to write that decoder purely from memory (no docs, no tools, no test-and-fix), Sonnet 4.6 got the byte order **wrong in 3 of 5 runs** — each failure emitting the same silently-corrupt value (the 16 wire bytes hexed in order, `e711b35c-…` instead of `61f0c404-…`). Opus 4.8 was reliable (5/5). The skill closes that gap: every run is correct by construction, no lookup required. The failure is invisible without a server to check against — the function returns a perfectly-formatted UUID string that just happens to be the wrong one.
